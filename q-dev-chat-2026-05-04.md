# FairHire AI — Complete Project Breakdown

---

## What Is This Project?

FairHire AI is an **AI-powered Applicant Tracking System (ATS)**. An ATS is the software companies use to manage the entire hiring process — from posting a job to sending an offer letter. This one uses Google Gemini AI to automatically read resumes, score candidates, and help HR teams make better hiring decisions faster.

Think of it as: **HR uploads 50 resumes → AI reads all of them → ranks candidates by fit → HR just reviews the top ones.**

## Backend Deep Dive

### How the server starts — `main.py` → `core/app_factory.py`

`main.py` is just 5 lines. It calls `create_app()` from `app_factory.py` which does the real work. Here's the startup sequence:

```
uvicorn main:app
       │
       ▼
create_app()
  1. register_middleware()  ← CORS, request ID headers
  2. register_routers()     ← attach all 12 route files to /api/v1/...
  3. lifespan() runs on startup:
       a. probe_postgres()  ← try to connect to PostgreSQL
       b. if fails → fall back to SQLite automatically
       c. create_all()      ← create all DB tables if they don't exist
       d. check JWT_SECRET  ← refuse to start if it's the default placeholder
```

**Why this design?** The DB engine is NOT created at import time. It's created after the probe decides which database to use. This means the same code works in dev (SQLite, no setup needed) and prod (PostgreSQL) without any code changes.

---

### `config.py` — The Settings File

Uses `pydantic-settings` which reads from `.env` automatically. Every environment variable the app needs is declared here with a default. If `JWT_SECRET` is still the placeholder string, the server crashes on startup with a clear error message — this prevents accidental insecure deployments.

---

### `db/models.py` — The Database Schema

Five tables, each as a Python class:

| Model | Table | What it stores |
|-------|-------|----------------|
| `HRUser` | `hr_users` | Recruiter accounts — email, hashed password, role |
| `Job` | `jobs` | Job requisitions — title, description, status, deadline |
| `Candidate` | `candidates` | People who applied — name, email, phone, resume text |
| `Application` | `applications` | Links a Candidate to a Job — stores all scores, stage |
| `Interview` | `interviews` | Scheduled interview sessions — round, date, score, feedback |

The `Application` model is the most important. It's the central record that tracks a candidate's entire journey through the hiring pipeline. It holds:
- `resume_score` — AI fit score from resume parsing
- `test_score` — score from the assessment
- `interview_score` — round 1 interview score
- `hr_interview_score` — round 2 interview score
- `final_score` — weighted composite of all above
- `stage` — where they are: applied → shortlisted → testing → interviewing → offered/rejected

`TextLengthMixin` is a Pydantic mixin that caps any free-text field at 20,000 characters to prevent someone from sending a 1MB string to Gemini and burning API quota.

---

### `db/session.py` — Database Connection

Creates the async SQLAlchemy engine. Uses `AsyncSession` so database queries never block the server. The `get_db()` function is a FastAPI dependency — every route that needs the DB gets a fresh session injected automatically.

---

### `db/probe.py` — Database Health Check

Before the app starts, this creates a throwaway connection to PostgreSQL, runs a no-op query, then immediately closes it. If it fails for any reason (wrong password, server down, wrong host), it returns the error string and the app falls back to SQLite. This is what makes the "zero config dev setup" work.

#### `services/parser.py` — Resume Text Extraction

This is the most complex file in the project. It handles the messy reality that PDFs are not structured documents — they're just positioned text on a page.

**The dual-parser strategy:**

```
PDF bytes
    │
    ├──► pypdf extracts text
    │         │
    │         ▼
    │    count section headings (Skills, Experience, Education...)
    │         │
    │    ≥ 2 headings? ──YES──► use pypdf output ✓
    │         │
    │         NO
    │         ▼
    ├──► pdfplumber extracts text (better at multi-column layouts)
    │         │
    │    compare heading counts, pick the better one
    │         │
    │    still < 2 headings? (graphic resume, icon-heavy)
    │         │
    │         ▼
    └──► Gemini AI cleans and restructures the text ✓
```

**The 6-pass cleanup pipeline** runs on the extracted text:
1. Unicode normalisation — fix encoding issues
2. Strip decorative symbols — box-drawing chars, dingbats, icon fonts
3. Restore missing spaces — camelCase splitting, space after commas (but first masks emails/phones so they don't get corrupted)
4. Rejoin broken words — PDF kerning sometimes splits "RELEVANT" into "RELEV ANT"
5. Isolate section headings — move inline headings to their own line
6. Whitespace collapse — clean up extra spaces and blank lines

**Why keep `raw_text_for_contacts`?** The cleanup pipeline adds spaces and modifies text. An email like `john@gmail.com` could become `john @ gmail . com` after some passes. So the raw uncleaned text is preserved separately just for email/phone/name extraction.

---

#### `services/profile_extractor.py` — Structured Data from Text

Takes the cleaned resume text and extracts structured fields.

**Skill extraction — 4-tier pipeline:**

```
Tier 1: Look inside the "Skills" section only
         → most reliable, skills explicitly listed
Tier 2: Look inside "Projects" section, specifically tech-stack lines
         → "Built with: React, Node.js, PostgreSQL"
Tier 3: Look inside "Experience" section
         → picks up skills mentioned in job descriptions
Tier 4: Full text fallback
         → catches anything missed above
```

Each tier adds to the result without duplicating. Skills are matched against the `SKILL_TAXONOMY` dictionary using compiled regex patterns.

**Contact extraction** uses a "contact zone" approach — it scans the first 25 lines of the resume (where name/email/phone always appear) rather than the whole document. It handles:
- Spaced emails: `john @ gmail . com` → `john@gmail.com`
- Obfuscated emails: `john [at] gmail [dot] com`
- Emoji prefixes: `📧 john@gmail.com`
- ALL CAPS names, hyphenated names, names with prefixes

**`contact_confidence()`** returns a score 0-100 based on what was found — email found = +40, phone = +40, name = +10, personal domain bonus = +10. This is returned in the API response so the frontend can warn if extraction quality was low.

#### `services/jd_matcher.py` — The Scoring Engine

This is the core AI scoring logic. It takes a candidate profile and a job description and produces a fit score 0-100.

**Four components run in parallel via `asyncio.gather()`:**

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  skill_score (20%)          experience_score (10%)      │
│  ─────────────────          ───────────────────────     │
│  Deterministic.             Deterministic.              │
│  Weighted taxonomy          Extracts years from JD      │
│  overlap. Skills in         ("5+ years required")       │
│  JD vs skills in            vs candidate's years.       │
│  resume. Each skill         Returns 0.0-1.0.            │
│  has a weight (Python       Runs synchronously.         │
│  = 1.5x, Figma = 1.0x).                                │
│  Runs synchronously.                                    │
│                                                         │
│  sem_score (30%)            impact_score (40%)          │
│  ────────────────           ──────────────────          │
│  Gemini embeddings.         Gemini analysis.            │
│  Converts profile to        Extracts achievement        │
│  natural language,          sentences from resume       │
│  embeds both profile        ("reduced latency by        │
│  and JD as vectors,         40%", "led team of 8").     │
│  computes cosine            Scores each sentence's      │
│  similarity.                business impact for         │
│  JD embedding is            THIS specific role.         │
│  cached (lru_cache).        Returns 0.0-1.0.            │
│                                                         │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
         fit_score = weighted sum
         - missing_skill_penalty (up to -10 pts)
         - clamped to 0-100
```

**Fallback redistribution:** If Gemini is unavailable (no API key, quota exceeded), `impact_score` and `sem_score` return 0.0. The weight of those components is redistributed proportionally to the remaining components so the total always sums to 100%. The score is lower but never crashes.

---

#### `services/semantic_matcher.py` — Gemini Embeddings

Converts text to a vector of numbers (an embedding) that captures semantic meaning. Two texts about similar topics will have vectors that point in similar directions — measured by cosine similarity.

The candidate profile is converted to natural language first:
```
"Candidate skills: Python, FastAPI, Docker.
 Education: Bachelor Of Technology, Computer Science.
 Experience: 3 years of professional experience."
```

This gives the embedding model enough context to understand the candidate semantically rather than just matching keywords.

The JD embedding is cached with `lru_cache(maxsize=64)` — when scoring 50 candidates against the same JD, the JD is only embedded once.

---

#### `services/scoring_service.py` — Impact Scoring

Sends achievement sentences to Gemini with a structured prompt asking it to score each sentence's business impact for the specific role on a 0-10 scale. Returns JSON with scores and the top 3 highlights.

Example: For a "Senior Backend Engineer" role, "Reduced API response time by 40% using Redis caching" scores 9/10. "Organised team lunch" scores 0/10.

---

#### `services/skill_taxonomy.py` — The Skills Dictionary

80+ skills, each with a list of aliases (all the ways people write the same skill) and a weight. The weight reflects how hard the skill is to find and how much it matters:
- `Kubernetes = 1.5` — rare, high value
- `Python = 1.5` — foundational, very common requirement
- `Figma = 1.0` — default weight

When computing skill overlap, matched skills are summed by weight, not count. A candidate with Kubernetes + Docker scores higher than one with Figma + Postman even if both matched 2 skills.

---

#### `services/decision_engine.py` — Hire/Hold/Reject

Takes all available scores and makes a structured decision:

| Composite Score | Skill Coverage | Decision |
|----------------|----------------|----------|
| ≥ 80% | ≥ 70% | Strong Hire |
| ≥ 65% | ≥ 50% | Hire |
| ≥ 50% | any | Hold |
| < 50% | any | Reject |

Interview score is weighted 1.2x (most reliable signal). Test score is 0.8x. Resume fit is 1.0x.

---

#### `services/email_service.py` — Email Notifications

Sends HTML emails at every pipeline stage. Uses Python's built-in `smtplib` with STARTTLS. Each email type has a hardcoded HTML template with the company branding. Falls back gracefully if SMTP is not configured — logs a warning instead of crashing.

## Key Design Decisions Explained

**Why FastAPI?** It's async-native, which means it can handle many concurrent requests without blocking. When one request is waiting for Gemini to respond, the server can handle other requests in the meantime.

**Why SQLite fallback?** So any developer can clone the repo and run it immediately without installing PostgreSQL. The same code runs in both modes — SQLAlchemy handles the dialect differences.

**Why `run_in_executor` for PDF parsing?** PDF parsing with pypdf/pdfplumber is CPU-bound (it does a lot of computation). FastAPI runs on an async event loop. If you run CPU-heavy code directly in an async function, it blocks the entire server — no other requests can be handled until it finishes. `run_in_executor` moves it to a thread pool so the event loop stays free.

**Why `asyncio.gather()` for scoring?** Semantic similarity and impact scoring both make network calls to Gemini. Running them sequentially would take 2x as long. `gather()` runs them concurrently — both API calls happen at the same time, and the function waits for both to finish.

**Why upsert instead of insert for candidates and applications?** If HR uploads the same resume twice (common when re-running with a better JD), you don't want duplicate records. The upsert pattern updates the existing record's scores instead of creating a new one. The candidate's pipeline stage is never reset — someone already in "Interviewing" stays there even if their resume is re-uploaded.

**Why hardcode `role = "hr"` on register?** If the role came from the request body, anyone could POST `{"role": "admin"}` and get admin access. The role is set server-side regardless of what the client sends.

**Why `TextLengthMixin`?** Gemini charges per token. A malicious user could send a 1MB string as a job description and burn thousands of dollars of API quota in one request. The 20,000 character cap prevents this.

- JD vector is cached so the same JD isn't re-vectorised for every candidate

## What Changed and Why

### `scoring_service.py` — Impact Scorer

**Problem 1: "3 projects" scored the same as "40% reduction"**
Fixed by splitting into `_STRONG_QUANT` (%, x, ms, users, revenue — real business units) and `_WEAK_QUANT` (projects, tasks, bugs — vague counts). Strong = +4pts, Weak = +2pts only if no strong quant exists.

**Problem 2: Padding resumes with filler sentences inflated scores**
Fixed by scoring the mean of ALL sentences, not just top-5. A resume with 20 mediocre sentences now scores lower than one with 5 strong ones.

**Problem 3: Filler penalty was too weak**
`-2` → `-3`. "Responsible for backend systems" is a genuine red flag in real ATS systems — it signals the candidate didn't own anything.

---

### `semantic_matcher.py` — TF-IDF Similarity

**Problem 1: Token length was a terrible IDF proxy**
"kubernetes" (10 chars) and "postgresql" (10 chars) got the same weight even though kubernetes is far rarer in job postings. Fixed with a real IDF table of 200+ tech terms built from actual job posting frequency analysis.

**Problem 2: BM25 term saturation**
The old TF was linear — mentioning "python" 10 times gave 10x the weight of mentioning it once. BM25 saturates: the 10th occurrence adds almost nothing. This prevents keyword-stuffed resumes from scoring higher.

**Problem 3: Institution name bias**
"B.Tech Computer Science IIT Bombay" and "B.Tech Computer Science State University" were producing different semantic vectors because "IIT" and "Bombay" are rare tokens with high IDF. Fixed by stripping institution names before vectorisation.

**Problem 4: Skills were repeated twice**
The old `build_profile_text` repeated skills twice to "boost TF weight". But the skill_score component already measures skill overlap — repeating in the semantic vector was double-counting the same signal.

---

### `jd_matcher.py` — Main Scoring Engine

**Problem 1: Missing data penalised freshers**
`experience_years = None` → `0.0` meant a fresh graduate with no date ranges in their resume got 0 on 10% of the score. Fixed: missing data returns `0.5` (neutral — we don't know, so we don't penalise).

**Problem 2: Education missing → 0.0**
Same issue. A candidate whose education section wasn't parsed got penalised. Fixed: returns `0.5` neutral.

**Problem 3: Hard experience cutoffs were unfair**
Old: 70% of required years → 0.8, 50% → 0.6, else → 0.3. This created cliffs. A candidate with 3.4 years for a 5-year role scored 0.3, while one with 3.5 years scored 0.6 — a 100% jump for 1 month difference. Fixed with a smooth `sqrt(ratio)` curve.

**Problem 4: Certifications penalised freshers**
Old: cert was a 15% weighted component. No certs = 0.0 on 15% of the score. A brilliant fresher with perfect skills but no AWS cert scored 15 points lower than a mediocre senior with one cert. Fixed: certs are now a **bonus** of 0–5 points, not a weighted component. No certs = 0 bonus, not a penalty.

**Problem 5: Linear missing skill penalty was unfair**
Missing 1 of 2 skills (50%) gave the same penalty as missing 5 of 10 (50%). But missing your only required skill is catastrophic, while missing 5 of 10 is manageable. Fixed with `sqrt(missing_ratio) * 0.12` — a concave curve that penalises heavily for the first few missing skills but plateaus.
